# Safar Go

## Overview

Safar Go is a premium luxury travel booking mobile app built with Expo (React Native) for the frontend and Express.js for the backend. It connects customers with drivers for inter-city travel bookings across Indian destinations (Ayodhya, Varanasi, Lucknow, Agra, Mathura, Prayagraj). The app operates as a car-hire/travel service with three user roles: customers, drivers, and admins.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

### Frontend (Expo / React Native)

- **Framework**: Expo SDK 54 with React Native 0.81, new architecture enabled, React 19
- **Routing**: expo-router with file-based routing. The `app/` directory contains route groups organized by user role:
  - `app/index.tsx` â€” Welcome/splash screen with auto-redirect based on authenticated user's role
  - `app/auth.tsx` â€” Combined login/registration screen
  - `app/customer/` â€” Customer flow with tab navigation (Home, Bookings, Profile), plus destination detail and booking screens
  - `app/driver/` â€” Driver flow with tab navigation (Dashboard, Earnings, Profile), plus KYC onboarding
  - `app/admin/` â€” Admin dashboard with tab navigation (Dashboard, Drivers, Manage)
- **State Management**: TanStack React Query for all server state. React Context for auth (`contexts/AuthContext.tsx`) and theme (`contexts/ThemeContext.tsx`). No Redux or Zustand.
- **Styling**: React Native `StyleSheet` with a custom theme system. **White/light theme is the default**, with dark mode available via toggle in Profile (persisted in AsyncStorage). Gold accent color (`#C5A55A`) is the brand color. Fonts are Playfair Display (headings) and Poppins (body) via `@expo-google-fonts`.
- **Bookings Screen**: Features a 2x2 quick actions grid (Bookings, Favorites, Support, Settings) at the top, followed by a "Recent Trips" section with booking cards.
- **Platform-specific components**: `MapSection` has separate `.native.tsx` (uses `react-native-maps`) and `.web.tsx` (uses a styled placeholder/fallback) implementations. The app handles web vs native differences throughout with `Platform.OS` checks.
- **Animations**: `react-native-reanimated` for enter animations (FadeInDown, FadeInUp)
- **Haptic feedback**: `expo-haptics` used on button presses and interactions (guarded by platform check for web)
- **Tab bars**: Support for native iOS tab bars via `expo-router/unstable-native-tabs` with `expo-glass-effect` detection, falling back to classic tab layout on other platforms

### Backend (Express.js)

- **Runtime**: Node.js with Express 5, TypeScript. Dev server runs with `tsx`, production build uses `esbuild`
- **Entry point**: `server/index.ts` â€” sets up CORS (dynamically based on Replit environment variables and localhost), JSON parsing, session middleware, and registers routes
- **Routes**: `server/routes.ts` â€” all API endpoints:
  - Auth: `/api/auth/register`, `/api/auth/login`, `/api/auth/logout`, `/api/auth/me`
  - Customer: destinations, bookings, favorites, reviews, coupons, wallet
  - Driver: KYC submission, profile, availability toggle, location updates, wallet/earnings
  - Admin: stats dashboard, user management, driver KYC approval/rejection, destination CRUD, coupon CRUD, settings
- **Authentication**: Session-based auth using `express-session` with `connect-pg-simple` (sessions stored in PostgreSQL). Passwords hashed with `bcryptjs`. No JWT â€” relies on session cookies with `credentials: 'include'` from the client.
- **Storage abstraction**: `server/storage.ts` implements an `IStorage` interface with all database operations using Drizzle ORM. All route handlers go through this abstraction layer rather than querying the database directly.
- **Auto-seeding**: The frontend triggers `POST /api/admin/seed` on app load to populate initial data.

### Database (PostgreSQL + Drizzle ORM)

- **ORM**: Drizzle ORM with PostgreSQL dialect
- **Schema location**: `shared/schema.ts` â€” shared between frontend and backend via path aliases (`@shared/*`)
- **Main tables**: `users`, `drivers`, `destinations`, `bookings`, `favorites`, `reviews`, `coupons`, `walletTransactions`, `driverWalletTransactions`, `adminSettings`, `activityLogs`
- **Schema features**: UUIDs via `gen_random_uuid()`, relations defined with Drizzle's `relations()`, insert schemas generated with `drizzle-zod`
- **Migrations**: Managed via `drizzle-kit push` (schema push approach, not migration files). Config in `drizzle.config.ts`
- **Connection**: `server/db.ts` creates a `pg.Pool` from `DATABASE_URL` environment variable

### API Communication Pattern

- The client uses a custom `apiRequest()` helper in `lib/query-client.ts` that prepends the server URL (derived from `EXPO_PUBLIC_DOMAIN` env var), sets credentials to `include`, and handles error responses
- TanStack React Query's default `queryFn` is configured globally to use `getQueryFn()` which handles 401 responses by returning null (for auth checks) rather than throwing
- The server URL in development is constructed from Replit environment variables (`REPLIT_DEV_DOMAIN`)

### Build & Deploy

- **Dev mode**: Two processes â€” `expo:dev` for the Expo dev server and `server:dev` for the Express backend
- **Production**: `expo:static:build` runs a custom build script (`scripts/build.js`) that bundles the Expo web app, then `server:build` bundles the server with esbuild. The server serves the static web bundle in production.
- **Port**: The Express server runs on port 5000
